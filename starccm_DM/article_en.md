STAR-CCM+ Design Manager is a powerful way to automate design exploration, DOE, and optimization studies from a single project. In practice, the study is defined in a `.dmprj` file, and Design Manager then launches and orchestrates the required simulations based on a reference simulation.

In our case, the project is `industrialExhaust_optimization.dmprj`, and it relies on the reference simulation `industrialExhaust_referenceSimulation.sim`, both stored in the `exhaust_opti/` folder. Running in batch mode means everything is launched without a graphical interface: the project is submitted, executed, and monitored automatically from a Python script.

In this article, we will:
- explain how Design Manager consumes licenses,
- detail the batch launch command,
- present a simple script for running on a single Xeon node,
- and then a more advanced script for larger topologies.

## Licensing

To run a STAR-CCM+ simulation on Qarnot, you must grant Qarnot access to your license. For more details, please contact our team at <a href="mailto:support-compute@qarnot-computing.com" target="_blank">support-compute@qarnot-computing.com</a>.

From this point forward, we will assume that your license configuration with us is already complete.

### How Design Manager consumes licenses

Design Manager licensing behavior determines how many license features are used during a study. Here, the word “cost” refers to license consumption, not pricing.

The points below are aligned with the Siemens <a href="https://docs.sw.siemens.com/documentation/external/PL20250327063586548/en-US/userManual/starccmp_userguide_html/STARCCMP/GUID-7873A15A-2776-4096-8AE3-142F2B7A3BA5.html" target="_blank">Design Manager Licensing</a> documentation:
- to read or update the reference simulation, Design Manager requires either `1 ccmpsuite`, `1 ccmppower`, or `10 powerpre`
- the study setup phase does not consume a license
- some study types, including `Optimization`, `DOE`, `Robustness and Reliability`, require `1 innovatesuite`
- if the reference simulation uses In-cylinder features, an additional In-cylinder license is required on the Design Manager server side
- Design Manager does not work with `hpcdomains`

### Default mode

In default mode, the Design Manager server checks out a session license and shares it with the design simulations. The designs then consume their own compute licenses:
- `1` core license for each concurrent design
- parallel licenses for the processors used by each design

By default, compute licenses are first taken from `DOEtoken`, with fallback options depending on the available licensing setup.

### Pre-Allocation

In the scripts presented in this article, we use `-preallocpower`. This mode means that Design Manager checks out the required licenses in advance before launching the designs, from a single license pool.

This mode is useful when you want predictable license usage and want license checkout to remain centralized on the Design Manager server side.

### General Job Submission

The other important variant is `General Job Submission`, enabled with `-dmnoshare`. In this mode, Design Manager does not share licenses with the designs: each design starts as a standard STAR-CCM+ simulation.

This approach is especially relevant if you want to drive your runs with `ccmppower` or Power on Demand, while still using Design Manager to orchestrate the study.

## Understanding the Design Manager command

The batch launch relies on `starlaunch jobmanager`, which drives the Design Manager study and delegates the actual solving work to STAR-CCM+ simulations.

For the simple script on a single Xeon node, the useful command looks like this:

```bash
starlaunch jobmanager --command "starccm+ -batch industrialExhaust_optimization.dmprj -preallocpower -passtodesign -power -licpath $CDLMD_LICENSE_FILE -mpi openmpi -np 26" --slots 0 --resourcefile /job/mpihosts
```

In the advanced script, you can then move to:
- `-np 94` on a `96c` machine,
- or `-np 52` on `2x28c`, together with `-machinefile /job/mpihosts` and `-mpiflags '--mca btl ^openib,tcp --mca pml ucx --mca osc ucx'`, which are only required in multi-node mode.

Here is the role of the main elements:

- `starlaunch jobmanager`: launches the Design Manager coordinator that orchestrates the study and distributes designs.
- `--command`: provides the STAR-CCM+ command that the job manager will run for the designs.
- `starccm+ -batch industrialExhaust_optimization.dmprj`: opens the Design Manager project without a graphical interface.
- `-preallocpower`: enables Pre-Allocation licensing mode.
- `-passtodesign`: passes execution and licensing options to the simulations generated by Design Manager.
- `-power`: requests Power licensing mode for the solver runs.
- `-licpath $CDLMD_LICENSE_FILE`: indicates where to find the license server.
- `-mpi openmpi`: selects the MPI implementation used for parallel execution.
- `-np ...`: sets the total number of MPI processes.
- `-machinefile /job/mpihosts`: provides the list of allocated nodes. This option is only required in multi-node mode.
- `-mpiflags ...`: passes OpenMPI options adapted to multi-node execution. This option is also only required in multi-node mode.
- `--slots 0`: lets the job manager rely on the provided resource file.
- `--resourcefile /job/mpihosts`: gives the job manager the list of resources allocated to the job.

## Launching a task on Qarnot

There are two ways to launch a task on Qarnot:
- Via our web interface <a href="https://app.qarnot.com/my-simulations" target="_blank">HPC</a>
- Using a script with an SDK

The rest of this blog post focuses on launching with Python scripts for increased automation and speed.

## Prerequisites

Before launching a calculation with the Python SDK, a few steps are required:
- <a href="https://app.qarnot.com/register" target="_blank">Create an account</a>
- Retrieve your <a href="https://app.qarnot.com/settings/access-token" target="_blank">API authentication token</a>
- <a href="https://doc.tasq.qarnot.com/documentation/sdk-python/installation.html" target="_blank">Install the Qarnot Python SDK</a>
- Know your Qarnot profile, for example `starccm-qarnot`

> **Note**: In addition to the Python SDK, Qarnot also provides SDKs for <a href="https://doc.tasq.qarnot.com/documentation/sdk-csharp/" target="_blank">C#</a> and <a href="https://doc.tasq.qarnot.com/documentation/sdk-nodejs/" target="_blank">Node.js</a>, as well as a Command Line Interface (<a href="https://doc.tasq.qarnot.com/documentation/cli/man/ManIndex.html" target="_blank">CLI</a>).

## Results

You should now have an `industrial-exhaust-opti-out` folder in your local working directory if you use the simple script, or an `industrial-exhaust-opti-advanced-out` folder if you use the advanced script. The same output bucket is also available on the <a href="https://app.qarnot.com/my-simulations" target="_blank">HPC app</a> with all files generated by the study.

You can directly view certain images and log files generated by the simulation in your output bucket via the <a href="https://app.qarnot.com/my-simulations" target="_blank">HPC app</a>.

Your results will be stored in the output bucket defined in the script and can be retrieved in three ways:
- Via the web platform: download directly from the Bucket section.
- As shown in the Python script: using the <a href="https://doc.tasq.qarnot.com/documentation/sdk-python/api/compute/task.html#qarnot.task.Task.download_results" target="_blank">download_results</a> function.
- Or via one of the <a href="https://qarnot.com/documentation/manage-data-dedicated-ui" target="_blank">open-source S3 bucket management applications</a>.

For a large number of files or more than 5GB of data, we recommend using rclone (Linux) or Cyberduck (Windows). Both are <a href="https://qarnot.com/documentation/manage-data-dedicated-ui" target="_blank">open-source S3 management tools</a>.
